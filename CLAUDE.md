# Audio to Haptics

## Project Overview
Web app that converts audio into haptic (vibration) feedback patterns. Takes an audio URL, analyzes the waveform using Web Audio API, and generates a vibration pattern for the Vibration API. Built with React 19 + TypeScript + Vite 7. Intended to become a standalone npm library (`audio-to-haptics`).

## Tech Stack
- **Framework**: React 19 with React Compiler (babel-plugin-react-compiler)
- **Build**: Vite 7, TypeScript 5.9
- **Linting**: ESLint 9 (flat config)
- **Testing**: Vitest 4
- **No CSS framework** — plain CSS, inline styles, ugly-MVP intentionally

## Architecture & How It Works

### Audio Analysis Pipeline
1. **Fetch & decode**: `analyzeAudio(url)` fetches audio, decodes via `AudioContext.decodeAudioData()`, returns first channel's `Float32Array` (full-spectrum), plus `outputLatency`/`baseLatency`. `decodeAudioBuffer(arrayBuffer)` exported separately for raw bytes.
2. **Bucket trends**: `computeTrends(data, sampleRate, bucketSize)` splits into buckets (default 2646 samples ≈ ~60ms), computes min/max per bucket plus leftRms/rightRms (unused by vibration logic). O(1) index: `trends[floor(currentTime * sampleRate / bucketSize)]`.
3. **Vibration map**: `computeVibrationMap(trends, opts?)` returns `boolean[]` — sustain check first, then noise floor gate, then past-only neighbor comparison. Vibrates when ratio >= `opts.spikeRatio`.
4. **Chain data**: `computeChainData(trends, vibrationMap, opts)` walks the vibrationMap, groups consecutive `true` runs into chains. For each bucket stores: `chainEndTime` (end of its run), `chainIntensity` (avg `computeIntensity` across run), `chainLength` (bucket count of run). All three are parallel arrays indexed by bucket.
5. **Vibration pattern**: `trendsToVibrationPattern(trends, vibrationMap)` converts to Vibration API format `[vibrate, pause, ...]`. Used for the fire-and-forget Vibrate button only.
6. **Display**: ResultView shows per-chain rows with time range, bucket count, intensity %, MAX or PWM pattern. WaveformView shows intensity-colored overlays (cyan→yellow→red) where `vibrationMap[i]` is true.

### Vibration API Usage
- **RAF-synced (Play+Vibrate)**: Lives in `HapticEngine.attach()`. RAF loop reads `audioEl.currentTime`, computes `bucketIndex`. Fires once on chain entry (`_wasVibrating` false→true): short chains (`chainLength < shortChainBuckets`) fire as `[remainingMs]` (solid MAX pulse); long chains fire `intensityToPattern(remainingMs, chainIntensity, opts)` (PWM). Calls `vibrate(0)` on chain exit. Never re-fires mid-chain.
- **Fire-and-forget (Vibrate button)**: `navigator.vibrate(pattern)` fires the full `trendsToVibrationPattern()` array at once.
- Only works on Android Chrome and some mobile browsers — no desktop support.

### Intensity Simulation (PWM)
`intensityToPattern(durationMs, intensity, opts)` simulates partial motor amplitude via PWM:
- `clamped = max(opts.intensityFloor, intensity)` — floor prevents motor stalling
- `onMs = round(opts.cycleMs * clamped)`, `offMs = cycleMs - onMs`
- Returns `[onMs, offMs, onMs, offMs, ...]` filling `durationMs`, last offMs absorbs remainder
- Motor inertia smooths 20ms cycles into perceived partial amplitude
- Default: `intensityFloor=0.5` (50% = `[10,10]`), `cycleMs=20`

### UI Components
- **App**: URL input with dropdown, Analyze/Play/Play+Vibrate buttons, VibrationTester section. Uses `HapticEngine` via `engineRef`. Snapshots engine state into `Analysis` interface for React rendering.
- **WaveformView**: Canvas waveform (0–1). 1-second window with navigation. Intensity-colored overlays (cyan=low, yellow=mid, red=high). Dashed noise floor line, white playhead bar (auto-advances). Three real-time indicators: SOUND/silence, VIBRATING/no vibration, ACTUAL SOUND/actual silence.
- **ResultView**: Metadata (sample rate, duration, latency), per-chain list showing time range, bucket count, avg intensity %, MAX (orange, solid pulse) or PWM pattern (grey). Vibrate/Stop/Copy Test Data buttons.
- **VibrationTester**: Hardcoded test button for manually firing vibration patterns to verify Android API limits.

### Current Vibration Criteria (Path B — Past-Only Neighbor Comparison)
- **Vibration triggers** (`computeVibrationMap()`) — for each bucket:
  1. **Sustain check** (before noise floor): if previous vibrated AND `current.max >= prev.max * sustainLowerBound` AND `current.max <= prev.max * sustainUpperBound` → vibrate. Bridges decay tails; upper bound blocks rising sections from being sustained.
  2. **Noise floor**: `max < computeNoiseFloor()` → never vibrate. Floor = `max(peakAmplitude * vibrateThresholdRatio, vibrateThresholdMin)`.
  3. **Past-only comparison**: `pastAvg = mean(last neighborRadius buckets)`. Vibrate when `trend.max / pastAvg >= spikeRatio`. Isolated bucket (no past or past avg < 0.001) always vibrates.
- **Debug logging**: Each vibrating bucket logs `max`, `pastAvg`, `ratio`. Sustained buckets log separately.

### Configuration
All algorithm knobs via `HapticOptions` in `analyzeAudio.ts`:
```ts
interface HapticOptions {
  bucketSize: number            // samples per bucket (default: 2646 ≈ 60ms)
  vibrateThresholdRatio: number // noise floor as fraction of peak (default: 0.4)
  vibrateThresholdMin: number   // absolute minimum floor (default: 0.040)
  neighborRadius: number        // past buckets to average (default: 4 ≈ 240ms)
  spikeRatio: number            // ratio threshold vs past average (default: 1.5)
  sustainLowerBound: number     // min ratio of current to prev max to sustain (default: 0.75)
  sustainUpperBound: number     // max ratio of current to prev max to sustain (default: 1.01)
  shortChainBuckets: number     // chains shorter than this fire as solid MAX pulse (default: 4)
  intensityFloor: number        // minimum PWM intensity, prevents motor stall (default: 0.5)
  cycleMs: number               // PWM cycle period in ms (default: 20)
}
```
- `DEFAULT_OPTIONS` provides sensible defaults. `HapticEngine` constructor accepts `Partial<HapticOptions>`.
- All analysis functions accept optional `opts` parameter with `DEFAULT_OPTIONS` as fallback.

## Current Approach: Past-Only Neighbor Comparison + Sustain + PWM Intensity

### Why This Approach
Every prior approach used an *absolute* reference (fixed threshold, fixed frequency range, single frame). Music doesn't have absolute quiet — it's *relatively* loud and quiet. The neighbor comparison is essentially **edge detection for audio**: a kick drum in a wall of guitar stands out *locally* even when globally everything is loud.

**Past-only refinement**: Symmetric neighbors still required SPIKE_RATIO ~2.0 for music but ~1.5 for isolated sounds. With past-only, a flat wall-of-sound has ratio ~1.0-1.1; genuine spikes clear the threshold. SPIKE_RATIO=1.5 works for both.

**Decay tails**: Solved with `sustainLowerBound`: natural decays are geometric, so `current >= prev × 0.75` catches the whole tail regardless of absolute level. Scale-invariant, self-terminating. `sustainUpperBound=1.01` prevents rising sections from being sustained through.

**PWM intensity**: Chain average intensity mapped to duty cycle. Short chains (< `shortChainBuckets`) fire as solid MAX pulse — punchy sounds feel like impacts. Long chains use PWM — sustained sounds have texture. Motor inertia at 20ms cycles smooths the pattern into perceived partial amplitude.

## Key Decisions & Learnings

### Bucket Size
2646 samples (~60ms at 44100Hz). Too small (10ms) = motor can't spin up. Too large = imprecise timing. ~40-100ms is the sweet spot.

### Vibration Timing / Sync
Root cause of early haptics was browser latency, not algorithm — `audioEl.currentTime` advances before sound physically reaches speakers. Syncing everything to `audioEl.currentTime` (not `performance.now()`) fixed it.

### Audio Pipeline Latency
Mute window suppresses haptics for `outputLatency + baseLatency` ms after play/seek/pause. Device values: Old Samsung ~168ms/~171ms, iPhone ~0ms/~3ms, Xiaomi Pad 5 8-40ms/~4ms.

Previously `_lastInterruption` (mute window start) was set from the DOM `play`/`playing` event. On Android, these events can fire before the audio hardware pipeline is warm, causing the mute window to start slightly early and letting the first haptic slip through before sound reaches the speakers. This was only visible in native-controls usage (SimpleUsage/MediaUsage) — App.tsx avoided it by calling `attach()` after `await audio.play()`, which used the fallback `_lastInterruption = performance.now()` in `attach()` instead of the event listener.

Fix: removed the `play`/`playing` listener entirely. `_lastInterruption` is now set inside the RAF loop when it detects a paused→playing transition (`wasPaused` local flag, initialized `true`). This anchors the mute window to the same RAF frame that begins haptic execution — no DOM event timing mismatch. Works identically for both attach-before-play (native controls) and attach-after-play (programmatic) patterns.

### Sustain
Natural decays are geometric. `sustainLowerBound=0.75` is scale-invariant and self-terminating. `sustainUpperBound=1.01` stops rising sections from being sustained through (only true decay tails qualify). Runs before noise floor gate.

### PWM Intensity
Motor inertia at 20ms cycles produces perceived partial amplitude — web-haptics approach, validated on Samsung. `intensityFloor=0.5` keeps motor spinning (at 40% floor the motor barely fires on quiet audio). Short chains get solid MAX pulse because a single-bucket chain has no room for PWM texture.

### Android Vibration API Pattern Limit
Android caps pattern arrays at ~200 entries. A 49-bucket chain fires `[10,10] ×135` = 270 entries and truncates at ~1.8s in. Acceptable — nobody wants 10-second sustained haptics anyway.

## Failed Approaches (Reference)

### Discounted Mean / PREVIOUS_WEIGHT
Symmetric neighbors with discounting. Still required different SPIKE_RATIO for music vs isolated sounds. Past-only solves this structurally.

### Post-Processing (burst density cap — reverted)
Onset memory got logic backwards — protected decay tails, removed onsets. `postProcessVibrationMap()` commented out.

### Bass-Filtered Precomputed
Lowpass 350Hz. Music still triggered on sustained bass. Code commented out in `analyzeAudio.ts`.

### Real-Time FFT (`Fast_Fourier_Transform` branch)
All 5 variants failed. Frame-by-frame FFT loses temporal context (bucket shape) that pre-computed analysis captures.

### Main Branch (`shouldVibrate`) Fails for Music
Full amplitude wall-of-sound fires threshold constantly (276 haptics on Chop Suey). Path B solves this.

## HapticEngine Class (`src/audio/HapticEngine.ts`)

### API
```ts
const engine = new HapticEngine({ spikeRatio: 2.0 }); // Partial<HapticOptions>, all optional
await engine.analyze(url);                              // fetch + decode + compute all pipeline outputs
await engine.analyzeBuffer(arrayBuffer);                // for raw bytes (file input, drag-and-drop, etc.)
await engine.load(url, mediaEl, onTick?);               // convenience: analyze + attach in one call
engine.attach(mediaEl, (time) => updateUI(time));       // RAF vibration loop + pause/seek listeners
engine.detach();                                        // stops everything, removes listeners
engine.muted = true                                     // suppress all navigator.vibrate() calls
engine.toggleMuted()                                    // flip muted state
engine.opts                                             // get copy of current HapticOptions
```

### Getters (all return copies — no mutation leaks)
`channelData` (Float32Array copy), `trends` (structuredClone), `vibrationMap`/`chainEndTime`/`chainIntensity`/`chainLength`/`pattern` (spread), `opts` (spread — all primitive fields), primitives returned directly.

### RAF loop (chain-based firing)
- Fires `navigator.vibrate(pattern)` once when `_wasVibrating` transitions false→true
- Short chain (`chainLength < shortChainBuckets`): `[remainingMs]` — solid pulse
- Long chain: `intensityToPattern(remainingMs, chainIntensity, this._opts)` — PWM
- `remainingMs = max(bucketDurationMs, chainEndTime[bucketIndex] - currentTime)` — ensures full chain duration
- Fires `vibrate(0)` when `_wasVibrating` transitions true→false

### Private internals
`_opts`, `_rafId`/`_wasVibrating`/`_lastInterruption` (RAF state), `_audioEl`/`_onTick`/`_cleanup` (attach lifecycle).

## Lessons Learned
- Zoom out before optimizing — first-haptic-early is negligible in streaming even though it looks bad in short test clips.
- Frame-by-frame FFT loses temporal context. Pre-computed bucket analysis captures sound *shape*.
- Every prior approach used an absolute reference. Making it local (compare to neighbors) solved the music problem.
- Past-only is strictly better than symmetric: future neighbors hurt onset detection structurally.
- Separate onset detection from decay sustain. Past-only handles onsets; `sustainLowerBound` handles decay chains.
- When tuning parameters can't fix a formula for two different cases, the formula is answering the wrong question for one of them.
- Short chains need solid pulses, not PWM — a 1-2 bucket chain has no room for texture, it just needs impact.
- Motor inertia at 20ms cycles smooths PWM into perceived partial amplitude. Don't try sub-20ms cycles.

## Problem-Solving Pattern: How the Percentage Trick Was Found

FORWARD_WEIGHT → PREVIOUS_WEIGHT → discounted mean — three attempts to fix decay tails via the ratio formula. All failed. The signal: when you keep adding parameters and one case still breaks, the formula is **answering the wrong question** for that case.

**Break out**: (1) Name the two behaviors separately. (2) Ask what each actually needs. (3) Look at the physical shape — natural decays are geometric, so a percentage threshold is scale-invariant and self-terminating. (4) Implement separately.

**General principle**: When one mechanism fails for two different cases, separate concerns, reframe each as its own question, look at the natural signal shape to find the right comparison.

```ts
})

engine.muted = true               // suppress vibration without stopping RAF
engine.toggleMuted()              // flip muted state
engine.opts                       // read current HapticOptions
engine.sampleRate                 // samples/sec of analyzed audio (typically 44100 or 48000)
engine.playbackIntensity          // same as onTick intensity — readable between frames
engine.playbackIsShortBurst       // same as onTick isShortBurst — readable between frames
```

```ts
// playbackIntensity: 0–1. 0 = silent or paused, 1 = loudest peak in the audio
```

### Haptic Visualizer (Trap Nation-style blob)

The library exposes `playbackIntensity` and `playbackIsShortBurst` specifically to drive audio-reactive visuals. These are the only two values you need. Use whatever rendering approach suits the design — SVG, canvas, WebGL, CSS — the library doesn't care.

**The two states:**
- `isShortBurst=true` → sharp transient hit (kick, gunshot, heartbeat). Animate aggressively: spike the geometry, flash bright, snap back fast. Intensity drives how strong the spike is.
- `isShortBurst=false` → sustained section or silence. `intensity` tells you how loud: 0 = silent/paused (return to rest), >0 = animate gently with organic breathing/undulation.

**How the data flows in React:**

Both values are React state and update every RAF frame, so the component re-renders automatically while playing — no separate animation loop needed. `playbackTime` is also available and useful as a continuous phase driver to keep sustained animations moving even when intensity is constant.

```tsx
const { playbackIntensity, playbackIsShortBurst, playbackTime } = useHaptics(videoRef)
// plug these directly into whatever visual you're building
```

**One approach we explored (SVG blob) — treat this as an idea, not a template:**

Generate polar coordinates around a circle, push points outward by `intensity`, run through a smooth bezier path. For bursts: few large bumps (4–6), high amplitude, snaps immediately since path recomputes each frame. For sustained: many small points modulated by overlapping sine waves, with `playbackTime` as the phase so the shape slowly rotates even during a held note.

A `smoothPath` utility that produces a smooth closed curve through N points using quadratic bezier midpoints works well for this — each point becomes a control point, midpoints become curve endpoints, result is always smooth regardless of point distribution.

This approach was prototyped and confirmed the primitives work. The visual result was mediocre — the right data was there, the implementation just wasn't polished. Canvas or WebGL would likely produce a better result. Pick the right tool for the design.

**What makes the animation feel alive:**
- `playbackTime` as phase driver — without it, sustained sections are static blobs. Rotating the phase at ~0.6 rad/s gives slow organic movement.
- Differentiating burst vs sustained visually — if both look the same, the visualizer feels flat.
- Snapping immediately on burst (no easing/transition) — transient hits are short, easing makes them feel late.
